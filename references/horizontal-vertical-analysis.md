# Horizontal-Vertical Analysis

Use this file only when the research task needs both time-depth and current-comparison. It is an optional method, not the default workflow of this skill.

## When To Use

Use horizontal-vertical analysis for:

- products and tools
- companies and organizations
- technical concepts or protocols
- market categories
- public figures or career trajectories

Use it when the reader needs to understand both:

- how the object became what it is
- how it compares with peers or substitutes now

Do not use it when the user only needs a short answer, a single-document summary, a narrow factual lookup, or a lightweight opinion.

## Core Structure

### 1. Vertical Analysis

Reconstruct the object's development over time.

Cover:

1. Origin: background, founding context, technical or market premise, initial problem, key people, and industry environment.
2. Birth point: first release, founding, first proposal, or first public form.
3. Evolution: key versions, financing, team changes, strategic shifts, architecture changes, user milestones, partnerships, acquisitions, crises, or controversies.
4. Decision logic: why major choices were made, what constraints existed, why one path was chosen over another.
5. Causal story: avoid a dry timeline; connect events into a readable account of cause, constraint, turning point, and consequence.

### 2. Horizontal Analysis

Compare the object with current peers, substitutes, or adjacent cases.

First classify the comparison scenario:

- Scenario A: no direct competitor. Analyze why: new category, high barrier, small market, regulatory limits, or indirect substitutes.
- Scenario B: one or two direct competitors. Compare each deeply.
- Scenario C: three or more competitors. Select the most representative three to five and mention the rest briefly.

Possible dimensions:

- technical route, method, or underlying logic
- product form, business model, organization structure
- target users and use cases
- strengths and weaknesses
- pricing, resource input, and scale
- user reception and real usage
- ecosystem position
- trend, opportunity, and risk

The comparison should not become a parameter table in prose. Explain what each peer has become, why users choose it, and where official positioning diverges from actual usage.

### 3. Intersection Judgment

The final section should combine both axes and produce new judgment, not summarize earlier sections.

Answer:

- Which historical choices explain today's position?
- Which strengths have historical roots?
- Which weaknesses were created by earlier good decisions?
- How do peers' different origins explain current differences?
- What is the most likely path from here?
- What are the upside and downside scenarios?

## Writing Guidance

- Prioritize readability over a mechanical consulting report.
- Use narrative in the vertical section, but keep facts traceable.
- Use comparison in the horizontal section, but keep it explanatory rather than tabular.
- Make judgments, but mark speculation clearly.
- Avoid empty terms such as "empower", "closed loop", or "strategic foothold" unless they are being critically analyzed.

## Suggested Length

Adjust to complexity:

- vertical analysis: 6,000-15,000 Chinese characters for complex objects
- horizontal analysis: 3,000-10,000 Chinese characters
- intersection judgment: 1,500-3,000 Chinese characters

These ranges are guidance, not hard quotas. Depth and relevance matter more than even allocation.

## Output Pattern

Use:

1. vertical analysis
2. horizontal analysis
3. intersection judgment
4. source notes or reference appendix

If the broader report has a different structure, integrate this method into the relevant sections instead of forcing these headings into the final article.
