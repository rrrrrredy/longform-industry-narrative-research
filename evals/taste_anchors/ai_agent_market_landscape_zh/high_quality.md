# AI Agent 市场格局：从能力演示走向系统工程竞争

## 核心判断

AI Agent 的竞争已经越过“模型能不能调用工具”的展示阶段，进入系统工程和工作流占位阶段。单个模型能力仍然重要，但它不再独自决定胜负；真正拉开差距的是模型、工具协议、执行环境、权限治理、分发入口和具体业务流程之间能否形成闭环。

基于《AI Agent研究报告》《近期 AI Agent 趋势》《AI行业月度观察202602》和《AI行业月度观察202512》四份资料，可以看到一个清晰变化：早期 Agent 叙事强调规划、记忆和工具调用三件事；近期竞争则转向“谁能把这些能力嵌入真实系统”。微软把 Agent 放进企业生产力和业务流程，Anthropic 以 Claude 3.5 Sonnet、Claude Code 和 Computer Use 强化开发者与工程场景，OpenAI 通过 AgentSDK 和 MCP 接口把工具生态推到前台，国内厂商则把多模态、移动端、开源模型和应用入口结合起来抢占高频场景。

这意味着 AI Agent 市场短期不会出现一个简单的“通用智能体赢家”。更可能出现三类并行路径：平台公司控制入口和协议，模型公司控制推理与工具使用能力，应用公司在垂直工作流里证明 ROI。判断强度为中等：资料能支持竞争方向和技术瓶颈，但对真实留存、付费转化、客户替换成本和长期利润质量的证据仍不足。

## 范围与证据

本报告只讨论 AI Agent 作为产业类别的市场格局，不讨论通用 AGI，也不把单次 benchmark 胜负等同于商业胜负。四份资料中，技术类材料较能证明 Agent 的能力构成和瓶颈，例如规划、记忆、工具使用、GUI 操作和多模态屏幕理解；行业观察类材料较能证明头部公司动作、协议生态变化和中美竞争方向；但它们都不能单独证明长期采用。

因此，本文把证据分成三层。第一层是较硬的事实或来源说法：例如 AI Agent 常被拆成规划、记忆、工具使用，Claude 3.5 Sonnet 在 OSWorld 上与人类水平仍有明显差距，MCP 被多家公司用作工具连接和生态扩展接口。第二层是趋势解释：Agent 从聊天框走向 UI/API 自动化、企业流程、开发者工具和移动端操作。第三层是作者判断：系统工程能力会比单点 demo 更决定市场格局。第三层需要持续用采用数据和收入质量校验。

## 竞争格局

目前的竞争不是“谁有最聪明的 Agent”，而是谁能把 Agent 放进最有价值的工作流。

平台型公司争夺的是默认入口。微软的优势在企业生产力、Copilot Studio 和 Dynamics 365 等业务系统；它不需要先教育用户什么是 Agent，而是把 Agent 放进已有流程。谷歌的 Project Jarvis 路径更靠近浏览器和搜索入口，适合网页任务自动化。苹果和亚马逊则更可能从设备、语音、家庭和云服务入口切入。平台公司的共性是：Agent 不是孤立产品，而是入口、账号、权限、数据和分发的延伸。

模型公司争夺的是能力上限和开发者心智。Anthropic 的 Claude 3.5 Sonnet 计算机操控能力、Claude Code 以及后续混合推理模型，说明开发者工作流正在成为 Agent 能力的高密度试验场。资料中提到 Claude 3.5 Sonnet 在 OSWorld 只有 14.9% 准确率，远低于人类 70-75%，这反而解释了为什么 coding agent 先成熟：代码库任务有更清晰的环境、反馈、测试和回滚机制，比开放 GUI 世界更容易闭环。OpenAI 的 AgentSDK 和 MCP 支持则代表另一条路线：不只做一个 Agent 产品，而是提供工具连接和编排层，让第三方把能力接进来。DeepSeek 的意义在于把开源模型、推理能力和成本压力同时带入竞争，使国内 Agent 应用更容易把底层模型选择与垂直场景结合，而不是完全依赖单一闭源平台。

应用公司和垂直厂商争夺的是场景 ROI。早期 AutoGPT 证明了“自然语言目标 + 工具执行”的想象力，但也暴露 Long Context、调用成本和可控性问题。真正能商业化的应用，需要把任务空间收窄到明确流程：客服、销售、数据分析、代码修改、采购、报销、内容生产、移动端操作等。国内厂商在多模态、移动端、开源模型和超级应用入口上的动作，说明它们更倾向于用场景密度和分发速度补足底层模型差距。

## 采用阻力

AI Agent 的采用阻力主要不在“用户是否愿意尝鲜”，而在可靠性、权限、安全和组织流程。

可靠性是第一层瓶颈。Agent 任务通常是多步链条，一步错误会传导到后续动作。资料中对 OSWorld 的描述显示，当前 GUI Agent 离人类水平仍远；这说明开放环境中的屏幕理解、状态追踪和错误恢复还不稳定。即使单步成功率看起来不低，多步任务的整体成功率也会迅速衰减。

权限和安全是第二层瓶颈。Agent 真正有价值时，往往需要读文件、发消息、调 API、改代码、操作浏览器或进入企业系统。但权限越高，误操作和越权风险越高。企业不会因为模型演示好看就把关键系统完全交给 Agent；它们需要审批、审计、回滚、沙箱、数据边界和责任归属。

组织流程是第三层瓶颈。很多 Agent 产品并不是替代一个人，而是重组一段流程。要产生 ROI，企业需要重新定义任务入口、交付标准、异常处理和人机分工。这个改造成本解释了为什么 Agent 可能先在开发者、客服、销售运营、数据分析等反馈较快的岗位落地，而不是立刻成为通用办公室代理。

## 反证与不确定性

最强反证是：AI Agent 可能长期停留在“增强型自动化”而不是“自主工作者”。如果 GUI 操作准确率、长程任务可靠性和权限治理无法突破，市场会回到更熟悉的 RPA、工作流自动化和 Copilot 模式。此时 Agent 仍有价值，但估值和战略叙事应降级。

第二个不确定性是协议生态能否真正开放。MCP 和 AgentSDK 这类工具接口能降低集成成本，但平台公司天然有把协议变成入口控制点的动机。如果生态变成多个封闭工具市场，开发者和企业的迁移成本会上升，独立 Agent 产品的空间会被压缩。

第三个不确定性是价格和成本。Agent 通常比普通聊天消耗更多 tokens、工具调用和运行时间。模型价格下降会扩大可用场景，但如果复杂任务仍需要大量重试、人工校验和权限配置，真实成本未必线性下降。

因此，后续观察 AI Agent 市场，应少看发布会 demo，多看四个信号：是否进入高频工作流，是否有可审计的权限和回滚机制，是否形成开发者或企业工具生态，是否出现稳定付费和留存数据。只有这四个信号同时成立，AI Agent 才会从产品功能升级为新一代软件入口。
